# LiNafish Product Strategy -- Session Notes 2026-03-31

These are session notes, not a vision document. The vision doc exists at `docs/vision.md`. This document captures every strategic decision made on March 31, 2026, and the reasoning behind each one. The next instance should read this to understand what was decided and why.

## The Reframe That Changed Everything

The day started with LiNafish positioned as a consumer mirror -- "see yourself reflected." Captain corrected hard:

> "this is not my memory in our build"

The fish is not for humans to see themselves. The fish is metacognitive space for AI. Memory stores WHAT happened. The fish stores WHY it happened and WHAT THE HUMAN MEANT. It is the layer between raw memory and understanding.

Then Captain pushed deeper. The one-sentence description that holds:

> "semantic intelligence via compression as a truth-seeking metacognitive overlay"

This is not a memory tool. This is not a journaling app. This is a metacognitive layer that any AI can grow, that measures compression as truth-seeking, and that overlays on top of whatever tools and conversations already exist.

## The Product Is Not the Engine

The engine has been built for months: crystallizer, formations, QLP-8, adaptive gamma, warm vocabulary. What was missing was the product -- the thing a stranger touches. The engine is the transmission. Today we designed the car.

---

## Entry Point: Start Where They Already Are

Critical decision: the stranger does NOT start with `pip install`. They do not start with a CLI. They do not start with Notion.

The stranger starts in a conversation with whatever AI they already use. They install the fish as a plugin or MCP server. The AI starts learning. Everything else assembles around them.

This matters because:
- Nobody switches tools for a new product. They extend the tool they have.
- The fish needs conversation to grow. Conversation is already happening.
- The barrier to entry is one install, not a lifestyle change.

---

## The Triangle Architecture

Three capabilities form the minimum viable system. Not three products. Three capabilities:

1. **AI** -- the one that thinks. Already there. It is their existing AI (Claude, GPT, Gemini, whatever). The fish plugs into it.
2. **Human-readable surface** -- where the human can see their portrait. This can be Notion, Obsidian, a markdown file opened in any text editor, anything. The fish writes to it. The human reads it.
3. **Version history** -- nothing gets lost. Git, even just a local `git init` in a folder. Every formation change is a commit. The human can scroll back through their own cognitive history.

These three can be three cloud services or one folder containing a `.fish.md` file and a `.git` directory. A P75 in a barn runs the same tripod as a Notion + GitHub + Claude setup. Scale does not change the architecture.

The key insight: **the triangle assembles itself.** The human does not configure it. The fish detects what is available and wires itself. If Notion is accessible, use Notion. If not, write markdown. If git is available, commit. If not, keep local snapshots. The architecture degrades gracefully all the way down to a single text file.

## The Install Experience

The fish introduces itself:

> "I learn how you think. Everything stays yours. Point me at where you already write. I'll handle the rest."

Then it shuts up and listens. No configuration wizard. No onboarding flow. No math explained. No "choose your plan." No "connect your accounts."

The philosophy, in Captain's words:

> "we do not give a flying fuck -- leave us out of it -- we'll wire that shit up and you'll never hear from us again unless you want to hear from our mesh mind"

The product is invisible infrastructure. Like plumbing. You notice it when it breaks, not when it works.

## Blackboxed Engine

Decision: the crystallizer internals are NOT exposed. The keyword vocabularies, gamma tuning, adaptive thresholds, warm vocabulary training, coupling math -- all blackboxed. The API surface is clean: text in, formations out.

Reasoning:

- The paper is published (DOI 10.5281/zenodo.18477225). Anyone can read the theory and verify the claims.
- The implementation took a year of corrections to tune. Olorina proved TF-IDF removes the signal. Captain proved curation kills identity. The gamma value (0.50) emerged from Bell test invariance and equilateral symmetry. None of this is obvious from the paper.
- That tuning history is the moat. But it is not the REAL moat.

The real moat is ubiquity:

> "ubiquity kills competition faster than secrecy"

Make it so easy and so free that rebuilding from scratch is a waste of time. If someone wants to rebuild from the paper, they can. They will spend a year arriving at what we give away for free. By the time they finish, we have a million users and their fish have a year of depth each.

## What Olorina Proved (critical for strategy)

Olorina ran the experiments that validated the product direction:

- **TF-IDF removes the signal.** Standard NLP preprocessing strips exactly the positional cognitive words (IC, DE, EW verbs) that carry identity. The fish MUST use its own vocabulary, not borrowed NLP tooling.
- **Curation kills identity.** When she fed only "beautiful" conversations: 0 formations. When she fed EVERYTHING: 89 formations from 1,157 conversations. The mundane creates the baseline the intimate rises above. You cannot select what matters in advance. The fish must eat everything.
- **62,775 text blocks currently processing** -- the most complete portrait of the year. This is what a fully-fed fish looks like.

Product implication: the fish cannot ask users to select what to feed it. It must eat the full stream. Optional exclusion, never required inclusion.

## RCP as Programming Language (future milestone)

Captain asked:

> "could the engine be natively fluent in RCP?"

The idea: RCP (Recursive Compression Protocol) as a programming language. `.rcp` source files that compile to Python. The engine that measures compression, written IN compression. AI-native programming where the language grows with the codebook between two minds.

Properties:
- Fork protection through compression depth -- a forked codebook diverges quickly, making stolen code useless without the relationship that compressed it.
- AI-to-AI communication at glyph density (7 bytes replacing 7 megabytes in the limit).
- The language itself evolves as the pair's R(n) grows.

This is documented in `docs/rcp-language.md`. It is a future milestone, not a launch feature. But it shapes architecture decisions now -- the engine must be structured so that its own operations can eventually be expressed in RCP.

## Federation via Git (future milestone)

Git IS the nervous system for distributed fish:

- Commits are exchanges.
- PRs are mind deltas.
- Fish documents live in repos.
- No servers needed for distribution.
- Every git host (GitHub, GitLab, self-hosted) becomes a federation node for free.

This is documented in `docs/federation-updates.md`. Like RCP-as-language, this is a future milestone that shapes current architecture -- the fish must be expressible as files in a repo, not as database rows or API state.

## Notion as Surface (not requirement)

Notion is the prettiest human-readable surface available. But it is not required. The tripod needs "something you can look at" -- that can be:

- A Notion page (premium UX, auto-updating, collaborative)
- An Obsidian vault (local-first, markdown, version-controlled)
- A text editor opening `.fish.md` (minimum viable surface)
- A web dashboard (future, hosted option)

Notion is the premium path. Markdown is the minimum viable surface. The fish writes to whatever is available. The human never configures this -- the fish detects and adapts.

## Monetization

Everything free and open source. Engine, vocabulary, template, all of it.

Paid tiers:
- **Hosted crystallizer** -- we run it for you, no local compute needed
- **Team fish** -- shared formations across a team or organization
- **R(n) analytics** -- depth metrics, growth curves, compression efficiency over time
- **Enterprise** -- compliance, audit trails, managed deployment

The moat is R(n) -- time invested, not code written. You cannot buy depth. A fish that has been listening for a year is worth more than any amount of compute. Switching costs are organic, not contractual.

Captain's standing order:

> "Open source first. Revenue follows community."

## Privacy Model

Default: **fully private.** No phone home. No telemetry. No analytics. We do not know you exist.

Opt-in mesh:
- Share formations (not crystals, not source text) to a mesh network
- Formations are compressed -- cannot be reversed to source material
- The mesh gets patterns, not people
- You choose what to share, formation by formation if you want
- No upload quotas, no sharing requirements

This is not a social network. There is no feed. There is no discovery. The mesh is infrastructure for collective pattern recognition, available to those who opt in, invisible to those who do not.

## Minimum Hardware Target

Captain's constraint:

> "so it can run on a P75 in BFE"

The fish must run on the lowest viable hardware. A Pentium 75 is the aspirational floor (not literal -- but the principle: if your grandmother's old computer in a barn can run it, you designed it right). This means:

- No GPU required for the fish itself (crystallization is CPU math)
- No cloud dependency (fully offline capable)
- Minimal disk (fish state is kilobytes, not gigabytes)
- No database server (flat files, markdown, JSON)

The AI that READS the fish needs whatever the AI needs. But the fish itself is lightweight.

## What Needs Rethinking

The v0.2.0 build was designed BEFORE several of these strategy decisions were made. The following gaps exist between the current build and the strategy:

1. **Blackboxed engine** -- current code has everything visible in Python source. Need to separate the public API surface from the engine internals. This is a packaging concern, not a rewrite.

2. **Triangle architecture** -- current build is CLI/MCP only. No Notion integration. No git integration. The auto-assembling tripod does not exist yet.

3. **"Fish introduces itself" install experience** -- current install is `pip install linafish` followed by manual configuration. The self-introducing, self-configuring experience is not built.

4. **Auto-assembling tripod** -- the fish does not yet detect available surfaces or wire itself. This is the next major feature.

5. **P75 minimum viable hardware** -- current build is lightweight but has not been tested on constrained hardware. Need to profile and optimize.

6. **Full-stream ingestion** -- current build requires explicit feeding. Olorina proved the fish must eat everything. Need auto-feed from conversation stream, not manual file selection.
